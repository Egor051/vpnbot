# HTTP-эндпоинт all-in-one подписки

**Отдельный процесс** (`python -m subscription_server`, юнит
`deploy/vpn-bot-subscription.service`) с единственным маршрутом — `GET /sub/{token}`, — который
отдаёт base64-подписку бандла: все его дочерние ключи в виде обычных клиентских ссылок. По
умолчанию **выключен** (`SUBSCRIPTION_ENABLED=false` — тогда на любой запрос отвечает `404`), UI
в боте пока нет.

Канонические справочники лежат там же, где и для остальных фич:

- **Переменные окружения** → [Конфигурация → All-in-one подписка](configuration.ru.md#all-in-one-подписка).
- **Семантика бандлов** (состав, отзыв, удаление) → тот же раздел.

## Как это работает

1. Клиент запрашивает `https://<host>:<публичный-порт>/sub/<token>`; токен — это секрет бандла
   (`key_bundles.token`, 256 бит, URL-safe) и **единственный** креденшл.
2. Эндпоинт читает `vpn.db` **только на чтение, вживую, через обычные репозитории** — без кэша.
   Поэтому отзыв, ротация токена или удаление действуют уже на следующем запросе, без рестарта,
   и эндпоинт продолжает работать, пока `vpn-bot.service` лежит.
3. Каждый ACTIVE-ребёнок ACTIVE-бандла рендерится **тем же кодом, что и одиночная выдача ссылки**
   (`XrayService._build_vless_link` для VLESS TCP/HTTP, `bot.formatters.format_hysteria2_link`
   для Hysteria2), ссылки склеиваются через `\n` и кодируются в base64. Ничего другого в
   подписку не попадает: AWG и прокси SOCKS5/MTProto исключены из состава бандла, поэтому строка
   любого другого типа роняет рендер целиком, а не молча пропускается.

### Fail-closed

Любой отказ — это **один и тот же пустой `404`**: неизвестный токен, отозванный бандл, удалённый
бандл, бандл без активных детей, битая строка ребёнка, нечитаемая БД и выключенный флаг. Отличить
никогда не существовавший токен от отозванного нельзя, частичный конфиг не отдаётся никогда.
`5xx` эндпоинт не отдаёт вообще: неожиданная ошибка перехватывается и превращается в тот же
`404`, так что трейс наружу не уходит. Единственный другой статус — `429` (с `Retry-After`) от
рейт-лимита, который срабатывает до обращения к БД.

### Заголовки ответа

| Заголовок | Значение |
|---|---|
| `Profile-Title` | Метка бандла (`bundle_XXXXX`); в `base64:` заворачивается только не-ASCII метка, отредактированная руками. |
| `Profile-Update-Interval` | `SUBSCRIPTION_UPDATE_INTERVAL_HOURS` (часы). |
| `Subscription-Userinfo` | `upload=`/`download=` — сумма реально собранных ботом счётчиков трафика детей (если не измерялось — часть опускается), `expire=` — общий `expires_at` детей в unix-секундах (если бессрочно — опускается). Если нет ни того, ни другого, заголовок не отдаётся вовсе. |
| `Cache-Control` | `no-store` — тело ответа это живой набор креденшлов. |

`total=` не отдаётся **никогда**: квоты по трафику в этом деплое нет, любое число здесь было бы
выдумкой, а клиенты читают выдуманную квоту как жёсткий лимит.

## TLS-терминация

**TLS терминирует сам процесс** (`ssl_context` на публичном порту). Reverse-proxy в стеке нет, и
этот PR его не вводит: nginx ради одного маршрута — это ещё один привилегированный демон, ещё
одна поверхность конфигурации и ещё одна зависимость по рестартам.

- **Публичный порт** — `SUBSCRIPTION_PUBLIC_PORT` (`0` — выключен, эндпоинт только на loopback).
  TCP/443 занят Xray REALITY, TCP/8443 — MTProxy, поэтому берётся свободный порт (например,
  `2096`) и открывается трекаемым правилом (ниже).
- **Открытым текстом наружу — невозможно**: публичный порт без обеих TLS-переменных не даёт
  процессу стартовать (`Settings.validate_subscription_ready`), а loopback-хост валидируется как
  loopback-адрес.
- **Чей ключ и под кем читается** — процесс работает под непривилегированным пользователем
  **`vpn-bot`** (как и `vpn-bot-hy2-auth.service`) и читает **групп-читаемую копию уже
  выпущенного** Let's Encrypt-сертификата для домена сервера — того же, что `acme.sh` ставит для
  Hysteria2, но *не* сам `/etc/hysteria/key.pem`. Отдельная копия оставляет материал Hysteria2
  нетронутым и даёт процессу ровно один читаемый секрет:

```bash
sudo install -d -o root -g vpn-bot -m 0750 /etc/vpn-bot/tls
# Добавьте ВТОРУЮ цель в существующий вызов acme.sh --install-cert (hysteria-цель
# не трогать), чтобы обновления попадали и сюда и перезапускали юнит:
sudo acme.sh --install-cert -d anycastedge.duckdns.org \
  --fullchain-file /etc/vpn-bot/tls/sub-fullchain.pem \
  --key-file       /etc/vpn-bot/tls/sub-key.pem \
  --reloadcmd      "chown root:vpn-bot /etc/vpn-bot/tls/sub-*.pem && \
                    chmod 0640 /etc/vpn-bot/tls/sub-key.pem && \
                    chmod 0644 /etc/vpn-bot/tls/sub-fullchain.pem && \
                    systemctl restart vpn-bot-subscription"
```

Ключ читается **один раз при старте**, поэтому обновление подхватывается этим рестартом.
`ReadOnlyPaths` для него юниту не нужен: `ProtectSystem=strict` делает `/etc` read-only, а не
невидимым, и доступ реально ограничивают права на файл.

## Установка (drift, руками)

`scripts/deploy.sh` автоматически ставит **только** `vpn-bot.service`; как и любой другой юнит из
`deploy/`, этот печатается как drift и ставится оператором. Phase 1 дополнительно печатает
информационную строку: установлен ли юнит, активен ли он и слушаются ли настроенные порты — она
никогда не фатальна, потому что хост без этого эндпоинта — нормальное состояние.

```bash
# 1. .env: включить фичу и выбрать порты
#    SUBSCRIPTION_ENABLED=true
#    SUBSCRIPTION_BIND_PORT=8445          # loopback, НЕ 8443 (занят xhttp/mtproxy)
#    SUBSCRIPTION_PUBLIC_PORT=2096        # 0 оставляет только loopback
#    SUBSCRIPTION_TLS_CERT=/etc/vpn-bot/tls/sub-fullchain.pem
#    SUBSCRIPTION_TLS_KEY=/etc/vpn-bot/tls/sub-key.pem

# 2. Юнит
sudo install -m0644 deploy/vpn-bot-subscription.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now vpn-bot-subscription

# 3. Фаервол (трекаемое правило — никаких `ufw allow` руками)
sudo bash deploy/ufw-subscription.sh          # порт берётся из .env
#   ... и обратно:
sudo bash deploy/ufw-subscription.sh --delete

# 4. Проверка
systemctl status vpn-bot-subscription --no-pager
sudo ss -tlnp | grep -E '8445|2096'
curl -si http://127.0.0.1:8445/sub/definitely-not-a-real-token | head -1   # ожидаем 404
```

Переключение `SUBSCRIPTION_ENABLED` требует `systemctl restart vpn-bot-subscription`: процесс
читает `.env` при старте. Юнит остаётся активным в обоих случаях — с выключенным флагом он держит
loopback-сокет и просто отвечает `404`, поэтому его состояние не «мигает» вместе с флагом, — но
**публичный слушатель при выключенном флаге не поднимается вовсе**: порт, который умеет только
отдавать `404`, это поверхность атаки без пользы.
